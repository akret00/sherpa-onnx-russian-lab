# ТЗ: Минимальный аудио-датасет для регрессионного тестирования пайплайна обработки речи

## 1. Общие положения
**Назначение:** Создание минимального контролируемого набора данных (датасета) для быстрой проверки и отладки пайплайна: VAD → Диаризация → Распознавание речи (ASR).

**Ключевые принципы:**
- Воспроизводимость: Каждый тестовый аудиофайл генерируется автоматически по машиночитаемому рецепту.
- Минимальная ручная работа: Вся эталонная разметка (Ground Truth) — побочный продукт процедуры генерации.
- Простота и прозрачность: Стек инструментов — только Python, NumPy, PyYAML, ffmpeg. Без тяжелых фреймворков и сложных систем сборки.
- Расширяемость: Архитектура "сегменты → сцены" позволяет наращивать датасет, добавляя новых дикторов, реплики и сценарии.
- Область применения: Только для внутреннего регрессионного тестирования. Не является репрезентативным датасетом русского языка.

**Архитектурные решения:** [dataset_ADR.md](dataset_ADR.md)

**Что остаётся за рамками данного ТЗ:**
- Метрики и бенчмарк для оценки качества моделей. Это тема отдельного ТЗ.
- Инструменты для сбора и разметки реальных записей.

## 2. Процесс подготовки сегментов
Этот раздел описывает полный путь от записи голоса диктором до готового набора сегментов, пригодных для использования в рецептах сцен.

### 2.1. Запись исходного материала
Цель — получить по одному непрерывному аудиофайлу от каждого диктора.
- **Список фраз:** Готовится заранее, 30–80 фраз на диктора. Фразы должны быть нейтральными, удобочитаемыми, длиной от 3 до 10 слов. Примерный список фраз: [Тексты для записи (Скрипты)](dataset_reqs.md#Тексты%20для%20записи%20(Скрипты))
- **Процесс записи:** Диктор читает фразы по одной, с выражением и естественной интонацией. Между фразами выдерживается пауза 1.5–2 секунды.
- **Оборудование:** Запись ведётся на диктофон, смартфон или гарнитуру, в тихом помещении, без посторонних шумов. Например, запись в WAV (через `alsa-utils` в командной строке):
	`arecord -f S16_LE -r 48000 -c 1 speaker001.wav`
- **Результат:** Один аудиофайл на диктора в любом lossless-формате (например, WAV), который затем вручную конвертируется в Opus командой вида 
	`ffmpeg -i speaker001.wav -c:a libopus -b:a 32k speaker001.opus`
- **Минимальные требования (v0):**
    - 3–4 диктора с различающимися голосами. Рекомендуемый состав: 2 мужских, 2 женских.
    - По одному записанному файлу на диктора.

### 2.2. Автоматическая предразметка и ручная коррекция
Полученный файл необходимо разбить на отдельные сегменты.
- **Автоматический этап:** Запись прогоняется через инструмент распознавания речи с VAD/диаризацией. Система определяет границы речи по паузам длиннее ~1 секунды и формирует черновой набор сегментов с полями `speech_start`, `speech_end` и гипотезой текста.
- **Ручная коррекция:** Черновик правится человеком до приемлемого качества. На данном этапе качество оценивается субъективно ("на глаз/на слух"), критерий — точное совпадение границ начала и конца фраз с произнесёнными словами, отсутствие обрезанных слогов и захвата лишней тишины.
- **Результат:** Набор сегментов, готовых к добавлению в `segments.yaml`.

## 3. Дизайн данных
Датасет строится по двухуровневой модели: **сегменты (segments)** → **сцены (scenes)**.

### 3.1. Сегменты (исходные аудиофрагменты)
Атомарная единица датасета — короткая одноголосая реплика, заданная (указанная) разметкой в исходном файле диктора.
- **Содержание:** Одна законченная фраза или предложение.
- **Длительность:** 1–6 секунд.
- **Качество:** Чистая речь (без фонового шума, музыки, реверберации).
- **Формат хранения аудио:** Opus. Все реплики одного диктора хранятся в одном аудиофайле, физически не нарезаются на отдельные файлы. Границы сегментов описываются метаданными.
- **Файл метаданных:** `segments.yaml`
    ```yaml
    dataset_version: 0.1
    sample_rate: 16000
    speakers:
      - id: 1
        name: "alex"
        notes: "headset mic, same room"
      - id: 2
        name: "maria"
        notes: "laptop mic"
    files:
	  - id: 1
	    file_path: speakers/SPK01/SPK01.opus  # относительный путь к файлу диктора
    segments:
      - id: 1
        speaker_id: 1
        file_id: 1
        text: "Привет. Проверяем запись голоса, раз, два, три."
        speech_start: 0.20                 # секунды от начала файла
        speech_end: 2.70                   # секунды от начала файла
      - id: 2
        speaker_id: 1
        file_id: 1
        text: "Сегодня я говорю нормально и достаточно спокойно."
        speech_start: 4.50                 # пауза ~1.8 сек от конца предыдущей
        speech_end: 7.60
    ```

### 3.2. Рецепты и итоговые Сценарии
Для обеспечения вариативности и автоматического тестирования диаризации, физические сцены генерируются на основе **Рецептов** (абстрактных шаблонов) путем ротации реальных спикеров.

#### 3.2.1. Рецепты (Абстрактные шаблоны сцен)
Рецепт описывает структуру будущей записи в терминах «ролей» (дикторов) и логических фраз, не привязываясь к конкретным аудиофайлам.
- **Файл метаданных:** `recipes.yaml`
- **Спецификация полей:**
    - `recipe_id`: Уникальный строковый идентификатор рецепта.
    - `description`: Описание проверяемого аспекта пайплайна.
    - `pause_ms`: Глобальный межфразовый интервал в миллисекундах, добавляемый между репликами.
    - `timeline`: Список шагов генерации.
        - `step_id`: Порядковый номер шага.
        - `dictor_id`: Логический номер роли (абстрактный диктор `1..N`).
        - `phrase_id`: Идентификатор фразы из общей базы/словаря.

**Пример структуры `recipes.yaml`:**
```yaml
recipe_id: "R001_clean_sequential"
description: "Sequential phrases without overlap"
pause_ms: 1000
timeline:
  - step_id: 1
    dictor_id: 1
    phrase_id: 1
  - step_id: 2
    dictor_id: 2
    phrase_id: 2
  - step_id: 3
    dictor_id: 1
    phrase_id: 3
```

#### 3.2.2. Итоговый файл сценариев (Физическое описание сцен)
Сценарий является результатом трансляции рецепта под конкретный набор физических файлов и спикеров. Он содержит точные таймкоды для сборщика аудиофайлов.
- **Файл метаданных:** `scenes.yaml`
- **Спецификация полей:**
    - `dataset_version`: Версия структуры данных.
    - `sample_rate`: Целевая частота дискретизации аудио.
    - `scenes`: Список физических сцен (один рецепт порождает несколько сцен).
        - `id`: Уникальный ID сцены (строка вида `{recipe_id}_rot{N}`).
        - `description`: Текстовое описание с указанием маппинга ролей (например, `D1->Spk1`).
        - `events`: Список физических аудио-сегментов:
            - `file_id`: Идентификатор физического файла/спикера (соответствует индексу в списке `AudioFile`).
            - `segment_id`: Идентификатор конкретного сегмента из `segments.yaml`.
            - `start`: Рассчитанное время старта сегмента в итоговой сцене (в секундах, нарастающим итогом).
            - `gain_db`: Изменение громкости сегмента, в дБ. 0 - громкость без изменений.

**Пример структуры `scenes.yaml`:**
```yaml
dataset_version: 0.1
sample_rate: 16000
scenes:
  - id: R001_clean_sequential_rot1
    description: "Rotation 1: D1->Spk1, D2->Spk2, D3->Spk3, D4->Spk4"
    events:
      - file_id: 1
        segment_id: 1
        start: 0.00
        gain_db: 0.0
      - file_id: 2
        segment_id: 102
        start: 3.50
        gain_db: 0.0
```

### 3.3 Сцены
Сцена — итоговый микс из нескольких сегментов с заданными параметрами, имитирующий акустическую ситуацию. Собирается из сегментов по рецепту.
- **Формат аудио:** Opus в контейнере `.opus`.
- **Файл рецептов:** `scenes.yaml`
    ```yaml
    dataset_version: 0.1
    sample_rate: 16000
    scenes:
      - id: S001_clean_2spk_nooverlap
        description: "2 speakers, clean, no overlap"
        events:
          - file_id: 1
            segment_id: 1
            start: 0.00
            gain_db: 0.0
          - file_id: 1
            segment_id: 2
            start: 3.50
            gain_db: 0.0
          - file_id: 3
            segment_id: 3
            start: 7.20
            gain_db: 0.0
        noise:
          enabled: false
      - id: S002_overlap_500ms_cafe_snr10
        description: "Controlled overlap 0.5s + cafe noise SNR=10dB"
        events:
          - file_id: 4
            segment_id: 33
            start: 0.00
            gain_db: 0.0
          - file_id: 2
            segment_id: 34
            start: 1.80
            gain_db: 0.0
        noise:
          enabled: true
          file: assets/noise/cafe.opus
          snr_db: 10
    ```
- **Генерация эталонной разметки** (происходит автоматически при сборке сцены):
    - `ref/{scene_id}.rttm` — разметка дикторов в стандартном формате RTTM.
    - `ref/{scene_id}.txt` — сквозная текстовая расшифровка для сцен без пересечений (overlap). Создаётся конкатенацией текстов сегментов в порядке их появления. Для сцен с overlap этот файл не создаётся.

## 4. Сценарии тестирования (Минимальная матрица v0)
Набор базовых сценариев для проверки пайплайна.

| ID сценария | Описание                    | Особенность                                        | Примечание                                                   |
| ----------- | --------------------------- | -------------------------------------------------- | ------------------------------------------------------------ |
| **A**       | `2spk_clean_nooverlap`      | Два диктора, паузы между репликами >200 мс         | Базовый тест дикторизации и ASR                              |
| **B**       | `2spk_asymmetric_nooverlap` | Длинные монологи первого, короткие реплики второго | Тест кластеризации при асимметрии                            |
| **C**       | `3-4spk_clean_nooverlap`    | Все дикторы, без overlap                           | Проверка масштабирования                                     |
| **D**       | `2spk_overlap`              | Реплики пересекаются на 200 мс и 500 мс            | **Только для VAD/сегментации.** ASR-разметка не генерируется |
| **E**       | `2spk_noise`                | Сценарий A + фоновый шум (SNR 10 и 20 дБ)          | Проверка устойчивости к шуму                                 |

## 5. Алгоритм генерации сценариев на основе рецептов
Процесс автоматического формирования `scenes.yaml` выполняется промежуточным скриптом, который принимает на вход абстрактный рецепт (`Recipe`) и список размеченных аудиофайлов (`List[AudioFile]`).

### 5.1. Обработка отсутствующих фраз (Стратегия Backup)
Так как не все спикеры могут иметь в наличии абсолютно все фразы из `recipe.yaml`, вводится константа `BACKUP_PHRASE_ID` (фраза-заглушка). Если при генерации шага у целевого спикера отсутствует нужный `phrase_id`, алгоритм автоматически подставляет сегмент, размеченный этим резервным ID.

### 5.2. Матрица циклической ротации
Для того чтобы проверить устойчивость моделей диаризации к разным голосам в одинаковых сценариях, скрипт дублирует рецепт \(N\) раз (где \(N\) — количество доступных `AudioFile`/спикеров), циклически сдвигая соответствие абстрактных ролей (`dictor_id`) реальным спикерам:
- **Вариант (Ротация) 1:** Диктор 1 = Спикер 1 | Диктор 2 = Спикер 2 | Диктор 3 = Спикер 3 | Диктор 4 = Спикер 4
- **Вариант (Ротация) 2:** Диктор 1 = Спикер 2 | Диктор 2 = Спикер 3 | Диктор 3 = Спикер 4 | Диктор 4 = Спикер 1
- **Вариант (Ротация) 3:** Диктор 1 = Спикер 3 | Диктор 2 = Спикер 4 | Диктор 3 = Спикер 1 | Диктор 4 = Спикер 2
- **Вариант (Ротация) 4:** Диктор 1 = Спикер 4 | Диктор 2 = Спикер 1 | Диктор 3 = Спикер 2 | Диктор 4 = Спикер 3

### 5.3. Шаги алгоритма расчета таймлайна
Для каждой из 4 вариаций ротации скрипт выполняет расчет таймкодов нарастающим итогом:
1. Устанавливается начальный маркер времени `current_time = 0.00` секунд.
2. Для каждого шага (`step`) из `timeline` рецепта определяется текущий физический спикер по матрице ротации.
3. В индексе сегментов этого спикера ищется элемент `AudioSegmentMarkup` с нужным `phrase_id` (или `BACKUP_PHRASE_ID`).
4. Вычисляется длительность сегмента: `duration = end_time - start_time`.
5. В генерируемую сцену добавляется событие (`ScenarioEvent`), где время старта округляется до сотых: `start = round(current_time, 2)`.
6. Маркер времени сдвигается для следующего шага: `current_time += duration + (pause_ms / 1000.0)`.

Сгенерированный массив сценариев сохраняется в итоговый `scenes.yaml` для последующей физической сборки аудио-микшером.

## 6. Процедура сборки датасета
Осуществляется одним Python-скриптом (`build_dataset.py`).
**Вход:**
- `segments.yaml`
- `scenes.yaml`

**Процесс:** Для каждой сцены скрипт:
1. Проходит по списку `events`.
2. Для каждого события находит сегмент по `segment_id` в `segments.yaml`.
3. Вызывает `ffmpeg` (подпроцесс) для декодирования нужного участка Opus-файла диктора в PCM-массив: `ffmpeg -ss speech_start -t длительность -i файл_диктора.opus ...`.
4. Размещает PCM-массив в итоговом миксе согласно `start` и `gain_db`.
5. При наличии шума миксует с шумом, приведённым к заданному SNR.
6. Кодирует итоговый микс обратно в Opus-файл.
7. Параллельно формирует и сохраняет `.rttm` и `.txt` (если применимо).

**Технические требования:**
- Микширование в NumPy — простое сложение массивов. Нормализация для предотвращения клиппинга.
- SNR: соотношение RMS сигнала к RMS шума.
- Детерминированность: при необходимости seed задаётся в рецепте.

## 7. Инструментарий и зависимости
- Python 3.10+
- NumPy
- PyYAML
- FFmpeg (вызывается через `subprocess`)

### Формат RTTM
Файл RTTM состоит из строк, где каждое слово разделено пробелом. Каждая строка представляет одно событие (обычно речевой сегмент).

Основные поля (по порядку):
1. **Type**: Тип сегмента (всегда `SPEAKER` для диаризации).
2. **File ID**: Имя аудиофайла (без расширения).
3. **Channel ID**: Номер канала (обычно `1`).
4. **Turn Onset**: Время начала сегмента в секундах (от начала записи).
5. **Turn Duration**: Длительность сегмента в секундах.
6. **Orthography**: (Обычно `<NA>`) для транскрипции.
7. **Speaker Type**: (Обычно `<NA>`) тип говорящего.
8. **Speaker Name**: Уникальный идентификатор диктора (например, `spk001`).
9. **Confidence**: (Обычно `<NA>`) оценка уверенности системы.
10. **Signal Lookahead**: (Обычно `<NA>`) технический параметр.

Пример:
```txt
SPEAKER audio_record_01 1 0.50 2.30 <NA> <NA> person_A <NA> <NA>
SPEAKER audio_record_01 1 3.10 1.20 <NA> <NA> person_B <NA> <NA>
SPEAKER audio_record_01 1 4.50 5.00 <NA> <NA> person_A <NA> <NA>
```
