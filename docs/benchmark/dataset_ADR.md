# Архитектурные решения (ADR)

Решения зафиксированы для обеспечения концептуальной целостности датасета и процесса его сборки.

## ADR-001 — Датасет = регрессионный тест-сьют, а не «репрезентативная выборка русского языка»
- **Решение:** Строить датасет как набор стабильных сценарных тестов для сравнения версий пайплайна, а не как объективный бенчмарк точности.
- **Обоснование:** Малый объём исключает статистическую репрезентативность, но отлично ловит регрессии и ошибки интеграции.
- **Последствия:** Все метрики интерпретируются как «по этому тест-сьюту», а не как «качество модели в целом».

## ADR-002 — Двухуровневая модель данных: сегменты → сцены
- **Решение:** Хранить библиотеку атомарных реплик (сегментов) и генерировать сцены по рецептам.
- **Обоснование:** Резко снижает ручной труд, позволяет быстро создавать вариации (noise, SNR, overlap, комбинации дикторов) без повторной разметки.
- **Последствия:** Необходим формат рецепта (`scenes.yaml`) и скрипт сборки. Единовременные затраты окупаются гибкостью.

## ADR-003 — Сегменты v0 должны быть строго одноголосыми и чистыми
- **Решение:** Сегмент = один диктор, одна реплика, без чужих голосов, музыки и фонового шума.
- **Обоснование:** Чистые сегменты максимально переиспользуемы. Шум и реверберацию лучше добавлять контролируемо на уровне сцены.
- **Последствия:** «Реализм» достигается на уровне сцен (шумовые ассеты, реверб-пресеты), а не «грязными» сегментами.

## ADR-004 — Эталонный текст хранится в сегментах и используется для ASR-оценки
- **Решение:** Включить поле `text` в описание каждого сегмента.
- **Обоснование:** Позволяет быстро дебажить и рассчитывать WER/CER без ручной расшифровки аудио.
- **Последствия:** Для сцен с overlap ASR-метрики становятся неоднозначными, поэтому логически отделены (см. ADR-007).

## ADR-005 — Сложные условия добавляются на уровне сцен, параметризованно
- **Решение:** Шум, реверберация, частотные искажения применяются после микса по рецепту, с фиксированными параметрами (например, SNR ∈ {20, 10, 0}).
- **Обоснование:** Обеспечивает полную воспроизводимость и управляемость. Один и тот же материал создаёт сетку тестов. Легче найти источник деградации.
- **Последствия:** Требуется библиотека ассетов (файлы шума) и строгие пресеты в рецептах.

## ADR-006 — Ground Truth диаризации генерируется автоматически из рецепта
- **Решение:** Разметка «кто и когда говорил» (RTTM) вычисляется напрямую из таймлайна сборки сцены.
- **Обоснование:** Исключает дорогую и предвзятую ручную разметку. Рецепт даёт идеальный, непротиворечивый GT по времени. Принято в соответствии с разделом 2.2, где ручная коррекция касается только границ и текста сегментов.
- **Последствия:** Необходимо хранить и версионировать рецепты. Разметка «живых» записей без сегментов и сцен — отдельная задача.

## ADR-007 — Разделение сценариев на «ASR-оцениваемые» и «для диаризации»
- **Решение:** Метрики WER/CER считаются только на сценах без overlap. Сцены с overlap используются исключительно для оценки VAD и диаризации.
- **Обоснование:** При одновременной речи сложно определить однозначный эталонный текст и порядок слов, что снижает интерпретируемость метрик.
- **Последствия:** Тестовая матрица явно делится на две подматрицы, результаты не смешиваются.

## ADR-008 — Формат разметки диаризации: RTTM
- **Решение:** Генерировать `*.rttm` как основной и единственный формат GT для диаризации и сегментации.
- **Обоснование:** RTTM естественно поддерживает пересечения (overlap), широко распространён и обеспечит совместимость с внешними инструментами оценки в будущем.
- **Последствия:** Один раз потребуется освоить формат, но это самый устойчивый и однозначный выбор.

## ADR-009 — Минимальные зависимости: Python + NumPy + PyYAML + FFmpeg
- **Решение:** Не использовать тяжелые фреймворки (например, полный pyannote) для подготовки или оценки данных.
- **Обоснование:** Соответствует требованию простоты и быстрых итераций. Важнее контроль и воспроизводимость, чем «идеальные» метрики ценой сложного стека.
- **Последствия:** Некоторые стандартизированные метрики (например, DER как в md-eval) либо считаются приближённо (frame-based), либо добавляются отдельным скриптом позже.

## ADR-010 — Оценка диаризации v0: frame-based «DER-like» на NumPy
- **Решение:** Рассчитывать внутри датасета простые показатели (Miss / False Alarm / Confusion) путём дискретизации времени с фиксированным шагом (например, 10 мс) средствами NumPy.
- **Обоснование:** Это соответствует цели регрессионного тестирования и ограничениям по зависимостям. Легко реализовать и отлаживать.
- **Последствия:** Результаты не будут точь-в-точь как у pyannote.metrics, но будут стабильны и сравнимы между версиями твоего пайплайна.

## ADR-011 — Запись сегментов: «один длинный файл с паузами» и автоматическая нарезка (v0)
- **Решение:** Диктор записывает 30–60 реплик подряд в один файл с паузами 1.5–2 секунды. Сегменты извлекаются автоматически (VAD/детектор тишины) с последующей ручной коррекцией границ и текста.
- **Обоснование:** Самый практичный способ для новичка: минимум ручных операций, быстро масштабируется на нескольких дикторов. Принято в соответствии с разделом 2.
- **Последствия:** Необходимо дисциплинировать процесс записи (паузы, темп) и один раз подобрать порог для автоматической нарезки.

## ADR-012 — Хранение границ речи внутри сегментов
- **Решение:** В описании сегмента (`segments.yaml`) хранятся точные внутренние границы речи (`speech_start`, `speech_end`). Небольшие поля тишины до и после границ в исходном файле диктора допустимы.
- **Обоснование:** Паддинг удобен для нарезки и прослушивания, но GT по времени речи в сценах должен опираться на реальные границы, иначе метрики VAD/DER будут некорректны.
- **Последствия:** Вводится дополнительное поле в метаданных, но это критично для качества GT. Принято и отражено в структуре `segments.yaml`.

## ADR-013 — TTS синтетика — отдельный класс тестов (smoke)
- **Решение:** Синтезированные голоса (TTS) можно добавлять для быстрого расширения и smoke-тестов, но их результаты не смешивать с живыми голосами при настройке порогов и принятии решений о качестве.
- **Обоснование:** TTS часто даёт нереалистичную внутридикторскую вариативность и спектральные артефакты, что может привести к ошибочной настройке системы.
- **Последствия:** В матрице тестов TTS-сцены помечаются отдельно, как «не для настройки порогов под реальных людей».
